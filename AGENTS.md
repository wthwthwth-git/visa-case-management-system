# AGENTS.md

## 1. 项目背景

本项目是签证案件资料管理系统 V1，面向小型事务所。

系统目标是帮助事务所围绕单个签证案件收集客户资料、管理资料项、查看上传文件、进行内部审核，并记录关键操作。

本项目不是完整 CRM，不是客户门户平台，也不是自动化签证申请系统。

## 2. 当前阶段

当前阶段是 Phase 0：项目文档初始化。

本阶段只允许创建和修改文档，不允许实现业务代码。

已规划的基础文档：

- `docs/product-spec.md`
- `docs/status-rules.md`
- `docs/page-flow.md`
- `docs/acceptance-checklist.md`
- `AGENTS.md`

## 3. 工作原则

后续任何代理或开发者都必须遵守：

- 先读文档，再做设计或开发。
- 不扩大 V1 范围。
- 不提前实现被明确排除的功能。
- 保持客户访问流程简单。
- 保持内部端和客户展示边界清晰。
- 所有重要业务规则应先写入文档，再进入实现。

## 4. V1 必须遵守的业务规则

### 4.1 客户访问

- 客户不注册。
- 客户不登录。
- 客户通过案件专属 token 链接访问。
- token 只允许访问对应案件的客户资料提交页面。
- token 不授予内部管理能力。

### 4.2 模板和案件资料项

- 创建案件时，模板资料项必须复制到案件资料项。
- 案件资料项不能直接实时引用模板资料项。
- 模板后续修改不影响已创建案件。
- 案件资料项后续修改不影响模板。

### 4.3 资料项和文件

- 一个案件可以有多个资料项。
- 一个资料项支持多个文件。
- 文件上传不代表资料项审核通过。
- 资料项状态必须由资料项自己的状态字段或概念表示。

### 4.4 状态分离

- 案件阶段和资料状态必须分离。
- 案件阶段描述整个案件进度。
- 资料状态描述单个资料项的提交或审核情况。
- 文件状态不代替资料状态。

### 4.5 内部备注

- 内部备注不得显示给客户。
- 客户页面不得返回内部备注。
- 内部备注只服务事务所内部处理。

### 4.6 Timeline Event

重要操作必须记录 timeline event。

至少包括：

- 案件创建。
- 模板复制。
- 文件上传。
- 文件删除或替换。
- 资料项状态变更。
- 案件阶段变更。
- 内部备注新增或重要更新。
- token 重新生成或失效。

## 5. V1 禁止实现的内容

V1 不实现：

- AI。
- OCR。
- 多员工权限。
- 聊天。
- 支付。
- 手机 App。
- 客户注册。
- 客户登录。
- 复杂通知系统。
- 自动表格填写。
- 第三方签证系统集成。

如果有人要求加入以上能力，必须先更新产品规格和验收清单，并明确这是 V1 之外的新范围。

## 6. 文档优先规则

当需求不明确时：

1. 先补充或修改文档。
2. 再确认验收标准。
3. 最后才进入实现。

任何影响业务对象、状态、权限边界、客户可见信息或 timeline event 的变更，都必须同步更新文档。

## 7. 开发前检查

进入业务代码开发前，至少检查：

- `docs/product-spec.md` 是否定义了对应业务规则。
- `docs/status-rules.md` 是否定义了状态边界。
- `docs/page-flow.md` 是否定义了页面和流程。
- `docs/acceptance-checklist.md` 是否有可验收条目。
- 是否违反 V1 禁止实现内容。

## 8. 修改文档的规则

修改文档时：

- 保持范围清晰。
- 避免加入 V1 未确认功能。
- 对新增规则补充验收清单。
- 对状态变更补充 timeline event 要求。
- 对客户可见内容和内部可见内容保持明确区分。

## 9. 重要约束摘要

- 客户不注册、不登录。
- 客户通过案件专属 token 链接访问。
- V1 不做 AI、OCR、多员工权限、聊天、支付、手机 App。
- 模板创建案件时必须复制到案件资料项，不能直接引用模板。
- 一个资料项支持多个文件。
- 案件阶段和资料状态必须分离。
- 内部备注不得显示给客户。
- 重要操作必须记录 timeline event。
