# 签证案件资料管理系统 V1 产品规格

## 1. 项目目标

本项目面向小型事务所，用于管理签证案件中的客户资料收集、文件提交、内部审核和案件进度记录。

V1 的目标不是做完整 CRM，也不是做客户门户平台，而是提供一个足够清晰、可控、低学习成本的案件资料管理工具：

- 内部人员可以创建案件、维护资料模板、跟进资料提交情况。
- 客户可以通过案件专属 token 链接查看需要提交的资料，并上传文件。
- 系统可以记录关键操作，帮助事务所追踪案件处理过程。

## 2. 使用场景

典型使用场景：

1. 事务所接到一个新的签证委托。
2. 内部人员创建案件，填写客户基础信息、现有签证类型和申请签证类型。
3. 系统或内部人员根据签证类型和客户具体情况确定适用资料模板。
4. 系统将模板资料项复制成该案件自己的资料要求。
5. 内部人员将案件专属 token 链接发送给客户。
6. 客户无需注册、无需登录，通过链接进入客户 Portal 页面。
7. 客户按资料要求上传一个或多个文件。
8. 内部人员审核资料，更新资料要求状态，并留下内部备注。
9. 如出现入管追加材料，内部人员手动新增资料要求。
10. 重要操作被记录为 timeline event，形成案件操作记录。

## 3. 用户角色

### 3.1 内部人员

小型事务所的工作人员。V1 不区分多员工权限，不实现复杂角色系统。

内部人员可以：

- 创建和查看案件。
- 维护资料模板和模板资料项。
- 从模板复制生成案件资料要求。
- 自定义追加资料要求。
- 新增入管追加材料资料要求。
- 查看客户上传的文件。
- 更新资料要求状态。
- 更新案件阶段。
- 管理客户确认申请书版本。
- 添加内部备注。
- 查看 timeline event。

### 3.2 客户

签证案件的申请人或资料提交人。

客户特点：

- 不注册账号。
- 不登录系统。
- 通过案件专属 token 链接访问自己的案件 Portal 页面。
- 只能查看面向客户展示的资料要求、说明和已上传文件信息。
- 不能看到内部备注。
- 不能看到事务所内部管理页面。

## 4. V1 范围

V1 包含：

- 案件基础信息管理。
- 资料模板和模板资料项管理。
- 创建案件时从模板复制资料要求。
- 统一案件资料要求列表。
- 每个资料要求支持多个文件。
- 客户 token 链接访问。
- 客户上传文件。
- 内部审核资料要求。
- 事务所负责资料要求。
- 自定义追加资料要求。
- 入管追加材料资料要求。
- 客户确认申请书多版本记录。
- 案件阶段管理。
- 资料要求状态管理。
- 内部备注。
- Timeline event 记录。

## 5. V1 不做的内容

V1 明确不做：

- AI。
- OCR。
- 多员工权限。
- 聊天。
- 支付。
- 手机 App。
- 客户注册。
- 客户登录。
- 复杂通知系统。
- 自动表格填写。
- 与政府或第三方签证系统集成。

如果后续需要扩展以上能力，必须先更新产品文档和验收清单，再进入开发。

## 6. 核心业务对象

### 6.1 Case 案件

案件代表一个客户的一次签证服务。

建议包含的信息：

- 所属客户。
- 案件编号。
- 现有签证类型。
- 申请签证类型。
- 当前案件阶段。
- 创建时间。
- 更新时间。

### 6.2 Customer Access Token 客户访问 Token

客户访问 token 用于客户不注册、不登录访问案件 Portal 页面。

重要规则：

- 每个 token 只对应一个案件。
- V1 中同一个案件同一时间只允许一个 active token。
- 重新生成 token 时，旧 token 必须失效。
- 数据库不应保存明文 token，只保存 token hash。
- token 不授予内部管理能力。

### 6.3 Document Template 资料模板

资料模板代表某类签证或场景通常需要收集的资料清单。

重要规则：

- 模板是正式数据库模型。
- 系统约有 210 套模板，需要支持后台导入、编辑和版本管理。
- 模板用于创建案件时生成初始资料要求。
- 模板不应该成为案件资料要求的实时引用来源。
- 模板后续修改不影响已创建案件。
- 模板未来需要版本审计能力，用于追踪模板创建、更新、发布和归档。

### 6.4 Document Template Item 模板资料项

模板资料项代表模板中的单个资料要求来源。

重要规则：

- 模板资料项只作为复制来源。
- 创建案件时，模板资料项应复制为 `CaseDocumentRequirement`。
- 复制后案件资料要求拥有独立生命周期。
- 模板资料项修改不影响旧案件。

### 6.5 Case Document Requirement 案件资料要求

案件资料要求是某个案件中具体需要处理的资料要求。V1 使用统一模型表示客户资料、事务所资料、自定义追加资料和入管追加材料。

核心区分字段：

- `responsibleParty`：`customer` 或 `office`。
- `sourceType`：`template`、`custom`、`immigration_request` 或 `system`。
- `portalVisible`：是否在客户 Portal 展示。
- `portalDownloadable`：客户是否可下载相关文件。

重要规则：

- 从模板复制时，必须把模板资料项复制到案件资料要求。
- 案件资料要求创建后，应作为该案件自己的独立数据存在。
- 后续修改模板，不应自动影响已经创建的案件资料要求。
- 后续修改某个案件的资料要求，也不应影响模板。
- 入管追加材料不是模板生成，而是后台手动新增的案件资料要求，`sourceType = immigration_request`。
- 客户只能看到客户可见且属于自己案件的资料要求。

### 6.6 Document File 文件

文件是客户或内部人员上传到某个案件资料要求下的具体文件。

重要规则：

- 一个资料要求可以有多个文件。
- 文件属于资料要求。
- 文件不等同于资料要求状态。
- 文件移除时应记录移除操作者类型和移除原因。
- 删除、替换、追加文件都应被视为重要操作，并按规则记录 timeline event。
- 客户不得访问内部不可见文件。

### 6.7 Application Confirmation 客户确认申请书

客户确认申请书用于记录客户通过 token 对申请书或确认材料的确认。

重要规则：

- 客户确认申请书不要求客户注册或登录。
- 申请书可能存在多个版本，例如 v1、v2。
- 新版本上传后，旧版本应被标记为已替代或非当前版本。
- 客户确认申请书不代表案件已递交。
- 客户确认动作必须记录 timeline event。

### 6.8 Internal Note 内部备注

内部备注用于事务所记录审核意见、处理说明、风险点或客户沟通摘要。

重要规则：

- 内部备注不得显示给客户。
- 客户 token 页面不能返回内部备注字段。
- 内部备注只服务内部处理流程。

### 6.9 Timeline Event 操作记录

Timeline event 用于记录案件中的重要操作。

应记录的操作包括：

- 案件创建。
- 案件阶段变更。
- token 创建、重新生成或失效。
- 模板创建、更新或版本创建。
- 从模板复制资料要求。
- 资料要求新增。
- 文件上传。
- 文件删除或替换。
- 资料要求状态变更。
- 内部备注新增或更新。
- 客户确认申请书版本创建或确认。

## 7. 案件访问方式

客户通过案件专属 token 链接访问案件 Portal 页面。

规则：

- 每个 token 只用于访问对应案件的客户页面。
- V1 同一个案件同一时间只允许一个 active token。
- 客户无需注册。
- 客户无需登录。
- token 页面只展示客户应看到的信息。
- token 不应授予内部管理能力。
- token 重新生成时旧 token 必须失效。

## 8. 模板与案件资料要求规则

模板创建案件时必须复制到案件资料要求，不能直接引用模板。

原因：

- 每个案件可能需要微调资料要求。
- 模板后续更新不应污染历史案件。
- 案件资料收集过程需要稳定、可追踪。

复制时建议保留：

- 来源模板 id。
- 来源模板版本。
- 来源模板资料项 id。
- 原模板资料项名称。
- 原模板资料项说明。
- 是否必需。
- 默认负责方。
- 排序。
- 可接受文件类型说明。

复制后，案件资料要求拥有独立生命周期。

## 9. 资料要求与文件规则

资料要求用于表达“需要处理什么资料”，文件用于表达“已经提交或上传了哪些具体文件”。

规则：

- 一个资料要求支持多个文件。
- 资料要求可以没有文件。
- 资料要求可以有一个文件。
- 资料要求可以有多个文件。
- 文件上传不应自动等同于资料要求已通过审核。
- 内部人员应明确更新资料要求状态。

示例：

- 资料要求：护照首页。
- 文件：passport-front.jpg、passport-renewed.pdf。

## 10. 入管追加材料规则

入管追加材料指案件过程中由入管或相关审查方要求补充的材料。

规则：

- 入管追加材料不依赖模板。
- 入管追加材料不要求修改原模板。
- 入管追加材料应作为 `CaseDocumentRequirement` 手动新增。
- 入管追加材料的 `sourceType` 必须是 `immigration_request`。
- 如果需要客户补交，应设置为客户可见，并在客户 Portal 中形成明确补充要求。
- V1 不与政府或第三方签证系统集成。

## 11. 案件阶段与资料状态分离

案件阶段和资料要求状态必须分离。

案件阶段描述整个案件处于哪个业务进度，例如：

- 新建。
- 收集中。
- 审核中。
- 准备递交。
- 已递交。
- 已完成。

资料状态描述单个资料要求的收集或审核情况，例如：

- 未提交。
- 已提交。
- 需补充。
- 已通过。
- 不适用。

禁止用资料状态代表案件阶段，也禁止用案件阶段代表资料状态。

## 12. 内部备注规则

内部备注只对内部人员可见。

规则：

- 客户页面不得展示内部备注。
- 客户接口不得返回内部备注。
- 内部备注可以关联案件、资料要求、文件或申请书确认。
- 内部备注新增和重要更新应记录 timeline event。

## 13. Timeline Event 规则

重要操作必须记录 timeline event。

Timeline event 的目标：

- 让内部人员知道案件发生过什么。
- 辅助排查客户是否提交过资料。
- 辅助审计资料状态和案件阶段变化。
- 降低小型事务所内部交接成本。

Timeline event 建议包含：

- 事件类型。
- 关联案件。
- 关联资料要求，可为空。
- 关联文件，可为空。
- 操作者类型，内部人员、客户或系统。
- 操作摘要。
- 结构化元数据。
- 发生时间。

Timeline event metadata 禁止保存：

- 明文 token。
- signed URL。
- 护照号码等敏感原文。

## 14. 基础数据关系

建议关系：

- 一个案件有多个客户访问 token 历史记录，但同一时间只允许一个 active token。
- 一个案件有多个案件资料要求。
- 一个案件资料要求有多个文件。
- 一个案件有多个 timeline event。
- 一个资料模板有多个模板资料项。
- 创建案件时，模板资料项被复制成案件资料要求。
- 内部备注可以关联案件、案件资料要求、文件或申请书确认。
- 一个案件可以有多个申请书确认版本。

## 15. 成功标准

V1 成功标准：

- 小型事务所可以为每个签证案件建立资料要求清单。
- 小型事务所可以维护约 210 套模板及版本。
- 客户可以通过 token 链接提交资料。
- 客户无需注册或登录。
- 一个资料要求可以接收多个文件。
- 内部人员可以清楚区分案件阶段和资料要求状态。
- 内部备注不会泄露给客户。
- 重要操作有 timeline event 可追踪。
- 系统不包含 V1 明确排除的复杂能力。
